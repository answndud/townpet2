# TownPet Incident Runbook

기준일: 2026-02-19  
적용 범위: `app` (Next.js), PostgreSQL, Redis(Upstash), 인증/알림/피드 API

## 1. 목적
- 장애 발생 시 탐지 -> 완화 -> 원인 분석 -> 재발 방지까지의 표준 절차를 고정한다.
- 담당자 교체/야간 대응 상황에서도 동일한 품질로 복구할 수 있게 한다.

## 2. 장애 등급
- SEV-1: 전체 서비스 중단, 로그인/피드 핵심 기능 불능, 데이터 손상 가능성
- SEV-2: 핵심 기능 부분 장애(예: 댓글/알림 실패), 우회 가능
- SEV-3: 일부 페이지/비핵심 기능 장애, 사용자 영향 제한적

목표 복구 시간:
- SEV-1: 30분 내 1차 완화
- SEV-2: 2시간 내 1차 완화
- SEV-3: 영업일 내 처리

## 3. 즉시 대응 체크리스트 (0~15분)
1. 영향 범위 파악: 로그인/피드/게시글/댓글/알림 중 어디가 깨졌는지 확인
2. health 확인:
   - `curl -sS -i http://localhost:3000/api/health`
3. 최근 배포/설정 변경 확인 (코드, env, DB 마이그레이션)
4. 로그 확인:
   - 서버 로그에서 `requestId`, `error.code`, 스택 트레이스 확인
5. 임시 완화:
   - 특정 기능 비활성화(예: 고비용 쿼리, 외부 연동)
   - 트래픽 우회/재시작 등 안전 조치

## 4. 진단 절차

### 4-1. DB 장애 의심
- 증상: 500 급증, prisma connection 오류, 타임아웃
- 확인:
  - `docker compose ps`
  - `docker compose logs --no-color postgres | tail -n 200`
  - DB 연결 문자열/크리덴셜 확인
- 조치:
  - 커넥션 풀/DB 리소스 확인
  - 최근 마이그레이션 롤포워드/롤백 판단

### 4-2. 캐시/레이트리밋 장애 의심
- 증상: 쓰기 요청 실패, rate-limit 오탐 증가
- 확인:
  - Upstash 연결 실패 로그 확인
  - 메모리 fallback 동작 여부 확인
- 조치:
  - fallback 활성 상태 확인
  - 외부 캐시 장애 시 핵심 기능 우선 복구

### 4-3. 인증/OAuth 장애 의심
- 증상: 로그인 실패, provider callback 오류
- 확인:
  - `AUTH_SECRET`, provider key/secret, callback URL
  - `/api/auth/*` 요청 로그/응답코드
- 조치:
  - provider 콘솔 리다이렉트 URI/키 롤백
  - 일시적으로 이메일 로그인 경로 안내 강화

## 5. 백업/복구 절차

### 5-1. 백업 (PostgreSQL)
- 주기:
  - 일 1회 전체 백업(논리 백업)
  - 마이그레이션 직전 수동 백업 추가
- 예시:
  - `pg_dump -Fc "$DATABASE_URL" > backup_YYYYMMDD_HHMM.dump`

### 5-2. 복구
- 사전 확인:
  - 현재 DB 스냅샷 별도 백업(되돌리기 대비)
  - 복구 대상 시점/범위 명확화
- 예시:
  - `pg_restore --clean --if-exists --no-owner -d "$DATABASE_URL" backup.dump`
- 복구 후 검증:
  - `/api/health` 200
  - 핵심 쿼리 샘플 검증(피드/게시글/댓글/알림)
  - 관리자 정책/제재 테이블 무결성 확인

## 6. 장애 종료 기준
- 에러율/응답시간이 정상 범위로 복귀
- 사용자 핵심 플로우(로그인, 피드 조회, 글 작성, 댓글)가 정상 동작
- 임시 완화 조치의 영향/롤백 계획이 문서화됨

## 7. 사후 회고 (24시간 내)
- 필수 기록:
  - 발생 시각, 탐지 시각, 완화 시각, 완전 복구 시각
  - Root cause
  - 즉시 조치/근본 조치
  - 재발 방지 항목 + 담당자 + 완료기한
- `PROGRESS.md`에 장애 요약/결정 기록 추가
