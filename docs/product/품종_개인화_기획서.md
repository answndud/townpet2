# breed-personalization-prd.md
# 품종 기반 개인화/광고/커뮤니티 기능 PRD (v0)

기준일
- 2026-02-23

문서 목적
- 반려동물 품종(견종/묘종) 정보를 기반으로 프로필, 피드, 광고, 커뮤니티를 연결하는 기능의 MVP 스펙을 정의한다.
- 바로 구현 가능한 수준의 범위, 수용기준, 데이터 스키마 초안, 추천 로직 의사코드를 제공한다.

문제 정의
- 현재 반려동물 프로필은 `species` 문자열 중심이라 개인화 신호가 약하다.
- 사용자에게는 내 반려동물 맥락에 맞는 정보 탐색 비용이 크고, 운영 측면에서는 광고/커머스 타겟 정밀도가 낮다.

목표
- 보호자가 반려동물 품종/체급/생애단계를 쉽게 설정하고, 피드에서 관련 정보를 더 빨리 발견하게 만든다.
- 품종 기반 커뮤니티(질문/후기/공동구매 모집)의 모임 밀도를 높인다.
- 광고 노출의 연관성을 개선하되 필터버블/과도한 상업화/악용 리스크를 함께 통제한다.

비목표 (MVP 제외)
- 유전자 검사 기반 품종 판별 자동화
- 완전 폐쇄형 품종 전용 피드(전체 피드 분리)
- 의료 진단/처방 추천 기능
- 결제/에스크로가 포함된 공동구매 정산 기능

핵심 사용자 스토리
- 보호자는 내 프로필에서 반려동물 품종을 설정하고, 관련 글을 우선적으로 보고 싶다.
- 보호자는 같은 품종 보호자들과 경험/용품 정보를 빠르게 교환하고 싶다.
- 광고주는 특정 품종/체급/생애단계에 맞는 상품을 타겟팅하고 싶다.
- 운영자는 스팸/사기성 공구를 자동 정책으로 빠르게 억제하고 싶다.

구현 상태 메모 (2026-03-07)
- 반려동물 프로필 create/update UI가 species별 품종 사전 옵션을 표시하고, `혼종/미상/직접 입력` fallback과 함께 `breedCode`, `breedLabel`, `sizeClass`, `lifeStage`를 저장한다.
- `/profile`, `/users/[id]`에서 품종/체급/생애단계를 일관되게 노출하고, 품종 코드가 있으면 품종 라운지로 연결한다.
- `UserAudienceSegment`는 반려동물 프로필 변경 시 자동 재생성되며, `/api/profile/audience-segments`와 내 프로필에서 세그먼트 요약을 확인할 수 있다.
- `/feed`와 품종 라운지에서 `일반 추천/맞춤 추천` 토글을 제공하고, 맞춤 추천 사용 시 현재 적용 세그먼트 또는 프로필 기준 설명을 노출한다.
- 피드 personalized ranking과 광고 audience key는 `UserAudienceSegment`를 우선 사용하고, 세그먼트가 없을 때만 pet profile 신호로 fallback한다.
- `MIXED`/`UNKNOWN`은 specific breed로 과대평가하지 않고, 종/체급/생애단계/혼종 라벨 중심 fallback audience key와 설명 문구를 사용한다.
- `preferredPetTypes`는 personalized feed의 2차 신호로 연결되어, 품종/프로필 직접 신호가 약한 경우에도 선택한 커뮤니티와 post `petTypeId` 매치를 약한 boost로 반영한다.
- `/feed` 맞춤 추천 설명은 품종/프로필 기준 외에 `선호 커뮤니티` 2차 신호를 함께 노출하고, 프로필 신호가 부족하면 선호 커뮤니티 기반 fallback summary를 사용한다.
- preferred community의 `Community.tags`는 personalized feed의 3차 신호로 연결되어, post `type`, `reviewCategory`, `petType.tags`, `animalTags`와 겹치는 `산책`, `건강`, `사료` 같은 관심 태그가 약한 boost로 반영된다.
- `/feed` 맞춤 추천 설명은 `관심 태그`와 콘텐츠 카테고리 3차 신호를 함께 노출하고, 프로필 신호가 부족한 경우엔 관심 태그 기준 fallback summary를 사용할 수 있다.
- 최근 `PostReaction`의 `LIKE/DISLIKE`는 personalized feed의 4차 engagement 신호로 연결되어, 최근 반응한 커뮤니티/관심 태그가 약한 boost 또는 suppress로 반영된다.
- `/feed` 맞춤 추천 설명은 `최근 반응` 4차 신호를 함께 노출하고, 프로필 신호가 부족한 경우엔 최근 반응 기준 fallback summary를 사용할 수 있다.
- `/api/feed/personalization` 경로로 personalized feed 조회/게시글 클릭/광고 노출/광고 클릭을 일별 aggregate로 수집한다.
- `/admin/personalization`에서 최근 7/14/30일 personalized feed CTR, 광고 CTR, surface/source 요약, 상위 audience key를 운영자가 확인할 수 있다.
- `/admin/breeds`에서 `BreedCatalog` effective 목록과 DB override/custom entry를 운영자가 직접 관리할 수 있다.

기능 범위 (MVP)
1) 프로필 확장
- Pet 단위로 `species`, `breedCode`, `breedLabel`, `sizeClass`, `lifeStage`를 저장한다.
- 혼종/불명 사용자를 위해 `MIXED`, `UNKNOWN` 옵션을 제공한다.
- 품종은 `BreedCatalog` 기반 선택 UI를 기본으로 하고, 사전에 없는 경우에만 직접 입력 fallback을 허용한다.

2) 피드 개인화 가중치
- 기본 정렬(최신/반응)을 유지하고 개인화 점수를 가산한다.
- 동일 품종 > 동일 체급 > 동일 종 순으로 가중치를 다르게 부여한다.
- 전체 노출 중 최소 35%는 일반/탐색 콘텐츠로 유지해 편향을 완화한다.

3) 품종 라운지
- `Breed Lounge` 탭: 인기글, 질문, 후기, 공동구매 모집 템플릿 글 목록 제공.
- 초기에는 카테고리 기반 필터 + 추천 정렬로 구현(복잡한 그래프 추천은 후속).

4) 광고 타겟팅
- 타겟 키: `species`, `breedCode`, `sizeClass`, `lifeStage`, `interestTags`.
- 광고는 명확히 라벨링하고, 건강 민감정보(질병 등) 기반 타겟은 금지한다.
- 동일 사용자의 광고 노출 빈도 캡(예: 일 8회, 세션 3회)을 적용한다.

5) 안전/정책 가드레일
- 공동구매/거래성 글은 신규유저 제한/링크·연락처 제한/신고 임계 자동숨김 적용.
- 품종 라운지에도 기존 신고/제재 정책을 동일 적용한다.

상세 수용 기준 (Acceptance Criteria)
- 프로필
  - 사용자는 최소 1개 반려동물에 품종 선택 또는 혼종/불명을 저장할 수 있고, 사전에 없는 품종은 직접 입력 fallback으로 저장할 수 있다.
  - 저장 후 `/profile`, `/users/[id]`에서 품종/체급/생애단계가 일관되게 노출된다.
- 추천
  - 품종 설정 사용자의 피드 상위 20개 중 최소 8개는 품종/체급 연관 글이 노출된다(데이터가 충분한 경우).
  - 같은 조건에서 일반/탐색 콘텐츠가 최소 7개는 유지된다.
- 광고
  - 광고 응답 payload에 타겟 세그먼트 매칭 근거가 포함되고(내부 로그), UI는 광고 라벨을 노출한다.
  - 빈도 캡 초과 시 해당 세그먼트 광고를 더 이상 노출하지 않는다.
- 정책
  - 신규유저의 고위험 카테고리 작성 제한이 품종 라운지에서도 동일하게 동작한다.
  - 신고 임계치 도달 글은 자동 `HIDDEN` 전환 및 audit 로그가 남는다.

데이터 모델 초안 (Prisma)

현행 대비 변경 포인트
- `Pet.species` 문자열 필드를 enum 기반으로 정규화.
- 품종 사전 테이블(`BreedCatalog`)을 추가해 코드 기반 저장.
- 품종 관심 기반 추천/광고를 위한 사용자 세그먼트 테이블을 추가.

초안 스키마

```prisma
enum PetSpecies {
  DOG
  CAT
}

enum PetSizeClass {
  TOY
  SMALL
  MEDIUM
  LARGE
  GIANT
  UNKNOWN
}

enum PetLifeStage {
  PUPPY_KITTEN
  YOUNG
  ADULT
  SENIOR
  UNKNOWN
}

model BreedCatalog {
  id          String      @id @default(cuid())
  species     PetSpecies
  code        String
  labelKo     String
  aliases     String[]
  defaultSize PetSizeClass
  isActive    Boolean     @default(true)
  createdAt   DateTime    @default(now())
  updatedAt   DateTime    @updatedAt

  @@unique([species, code])
  @@index([species, labelKo])
}

model Pet {
  id          String       @id @default(cuid())
  userId      String
  name        String
  species     PetSpecies
  breedCode   String?
  breedLabel  String?
  sizeClass   PetSizeClass @default(UNKNOWN)
  lifeStage   PetLifeStage @default(UNKNOWN)
  age         Int?
  imageUrl    String?
  bio         String?
  user        User         @relation(fields: [userId], references: [id], onDelete: Cascade)
  createdAt   DateTime     @default(now())
  updatedAt   DateTime     @updatedAt

  @@index([userId, createdAt(sort: Desc)])
  @@index([species, breedCode])
  @@index([sizeClass, lifeStage])
}

model UserAudienceSegment {
  id              String   @id @default(cuid())
  userId          String
  species         PetSpecies?
  breedCode       String?
  sizeClass       PetSizeClass?
  lifeStage       PetLifeStage?
  interestTags    String[]
  confidenceScore Float    @default(0.5)
  updatedAt       DateTime @updatedAt

  @@index([userId])
  @@index([species, breedCode, sizeClass, lifeStage])
}
```

마이그레이션/호환 전략
- 1단계: 새 enum/필드 추가(기존 `species` 문자열과 병행 저장)
- 2단계: 백필 스크립트로 `species` 문자열 -> enum/품종코드 매핑
- 3단계: 읽기 경로를 신규 필드 우선으로 전환, 구필드 fallback 유지
- 4단계: 운영 검증 후 구필드 제거

추천 로직 의사코드 (MVP)

```ts
type FeedCandidate = {
  postId: string;
  createdAt: Date;
  engagementScore: number; // like/comment/save/view normalized
  species?: "DOG" | "CAT";
  breedCode?: string;
  sizeClass?: "TOY" | "SMALL" | "MEDIUM" | "LARGE" | "GIANT" | "UNKNOWN";
  tags: string[];
};

type UserContext = {
  species?: "DOG" | "CAT";
  breedCode?: string;
  sizeClass?: "TOY" | "SMALL" | "MEDIUM" | "LARGE" | "GIANT" | "UNKNOWN";
  interestTags: string[];
};

function scoreCandidate(item: FeedCandidate, user: UserContext, nowMs: number): number {
  const ageHours = Math.max(1, (nowMs - item.createdAt.getTime()) / 3600000);
  const recency = 1 / Math.sqrt(ageHours);
  const engagement = item.engagementScore;

  let personalization = 0;
  if (user.breedCode && item.breedCode === user.breedCode) personalization += 0.45;
  if (user.sizeClass && item.sizeClass === user.sizeClass) personalization += 0.2;
  if (user.species && item.species === user.species) personalization += 0.1;

  const sharedTags = item.tags.filter((t) => user.interestTags.includes(t)).length;
  personalization += Math.min(0.15, sharedTags * 0.03);

  return recency * 0.5 + engagement * 0.35 + personalization;
}

function buildFeed(candidates: FeedCandidate[], user: UserContext): FeedCandidate[] {
  const nowMs = Date.now();
  const scored = candidates
    .map((item) => ({ item, score: scoreCandidate(item, user, nowMs) }))
    .sort((a, b) => b.score - a.score);

  // diversity guardrail: keep at least 35% exploration items
  const personalized = scored.filter((x) => x.item.breedCode && x.item.breedCode === user.breedCode);
  const exploration = scored.filter((x) => !personalized.some((p) => p.item.postId === x.item.postId));

  return interleaveWithRatio(personalized, exploration, 0.65).map((x) => x.item);
}
```

알고리즘 운영 원칙
- 학습 가능한 모델 도입 전까지는 설명 가능한 규칙 기반 점수로 시작한다.
- 점수 구성요소(최신성/반응/개인화)는 내부 로그에 분해 저장해 디버깅 가능해야 한다.
- 개인화 점수 상한을 두어 저품질 콘텐츠가 과도하게 상단 노출되지 않게 한다.

API/서비스 초안
- Profile/Pet
  - `PATCH /api/profile/pets/:id`: 품종/체급/생애단계 수정
  - `GET /api/profile/audience-segments`: 개인화 세그먼트 조회
- Feed
  - `GET /api/posts?mode=PERSONALIZED`: 개인화 가중치 적용 피드
  - 응답에 `rankingReason`(internal-only toggle) 필드 옵션 제공
- Lounge
  - `GET /api/lounges/breeds/:breedCode/posts`
  - `POST /api/lounges/breeds/:breedCode/groupbuys` (템플릿 검증 필수)

보안/정책 체크리스트
- 품종 라운지 글 작성/댓글 작성에도 rate limit 적용
- 신규유저 제한(24h) + 링크/연락처 제한 재사용
- 신고 임계 자동숨김 + 관리자 audit trail 유지
- 의료/건강 카테고리 문구는 정보 제공 중심(진단 유도 문구 금지)

실험/지표 (출시 후 4주)
- 활성 사용자 대비 품종 설정 완료율
- 품종 관련 글 클릭률/저장률/댓글률
- D7 재방문율 변화(설정 사용자 vs 미설정 사용자)
- 광고 CTR/CVR uplift(비타겟 대비)
- 품종 라운지 신고율/자동숨김 비율

출시 단계
- Phase A(2주): 프로필 확장 + 피드 가중치 + 관측 지표 계측
- Phase B(2주): 품종 라운지 + 공동구매 템플릿 + 정책 연결
- Phase C(2주): 광고 세그먼트 + 빈도 캡 + A/B 실험

오픈 이슈
- user-level 클릭/저장/광고 반응 로그까지 합쳐 recency-weighted 5차 개인화 신호로 확장하는 설계는 후속 작업 필요
- 공동구매 거래 보호(에스크로/사기 분쟁 처리) 범위는 별도 정책 문서 필요
