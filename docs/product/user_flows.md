# user_flows.md (개선안) — 핵심 사용자 플로우 (MVP~Phase 확장)

> 목적: 신입 개발자/PM/운영자가 “무엇을 먼저 만들고, 어디서 리스크가 터지는지”를 한 번에 이해하도록 핵심 플로우를 **정상 흐름 + 예외/운영 + 데이터/상태**까지 포함해 정리한다.
> 원칙: ① Local/Global 모드 혼탁 금지 ② “작성 템플릿(구조화)”이 플로우의 중심 ③ Market/Care/L&F는 기능보다 **정책/증빙/제재**가 먼저.

---

## 0) 공통 규칙(모든 플로우에 적용)

### 0.1 상태 변경은 항상 서버에서

* 클라이언트는 UI/UX, 서버는 **검증(Zod) + 권한 + 레이트리밋 + 감사 로그**를 강제한다.

### 0.2 공통 가드(Guardrails)

* **Auth**: 로그인 필요/불필요 구간 명확화
* **Rate Limit**: 작성/검색/신고/업로드 별도
* **콘텐츠 필터**: 연락처/외부링크/비방/도배 탐지
* **운영 정책**: 신고 누적 → 자동 블라인드 → 운영자 큐

### 0.3 공통 데이터(최소)

* `User`, `Neighborhood`, `Post(type/scope/status)`, `Comment`, `Report`
* L2/L3 데이터(정밀 위치, 증빙 이미지 등)는 기본 비공개 + 접근통제(privacy 문서와 일치)

## 1) 온보딩 / 동네 설정 (Local 모드 진입)

### 1.1 정상 플로우

1. **OAuth 로그인**(카카오/구글)
2. **닉네임/프로필 설정**
3. **대표 동네 선택**

   * 검색(행정동) + 지도(대략 위치) 병행
4. (선택) **동네 인증 레벨 업그레이드**
5. **홈(Local 요약) 진입**

### 1.2 예외/운영

* 닉네임 중복 → 즉시 검증 + 대안 제시
* 동네 미선택 → Local 콘텐츠 접근 제한(“동네 선택 필요” 게이트)

### 1.3 개발 포인트

* `UserNeighborhood(isPrimary)` 설정 시나리오(대표 1개 보장)
* 인증 레벨은 Phase로 분리하되, **정책 엔진(가입/활동/인증)** 인터페이스는 미리 설계

---

## 2) 병원 리뷰(템플릿, 구조화 핵심)

### 2.1 정상 플로우

1. Local > 병원 > **리뷰 작성**
2. **병원 선택**

   * 검색(기존) / 직접 입력(신규)
3. 템플릿 입력(필수)

   * 방문일, 진료 항목 태그, 총 비용, 대기시간, 평점
   * (선택) 영수증 체크 + 항목별 비용(JSON)
4. **게시**
5. 사용자 액션: 저장/댓글/신고
6. 병원 상세(프로필)에서 **분포/중앙값/최근성** 기반 요약 노출

### 2.2 예외/운영

* 비방/명예훼손 키워드 감지 → 작성 단계 경고(soft block) + 게시 후 신고/자동 블라인드
* 영수증 이미지 업로드 시:

  * EXIF 제거, 민감 정보 마스킹(가능하면), 접근은 최소(privacy 정책)
* 악성 리뷰/업자 바이럴:

  * 신규 계정의 리뷰 게시 제한(쿨다운/활동 기준)
  * 동일 병원 반복 도배 탐지

### 2.3 데이터/상태

* `Post(type=HOSPITAL_REVIEW, scope=LOCAL, status=ACTIVE)`
* `HospitalReview(costBreakdown JSON?)`
* 병원 엔티티를 별도 모델로 분리할지(향후 Place와 통합) 결정 필요

---

## 3) 동반 가능 장소 리뷰(장소/조건 태그)

### 3.1 정상 플로우

1. Local > 동반가능 장소 > **리뷰 작성**
2. 장소 선택/등록(지도 기반 추천 가능)
3. 조건 태그(필수 최소)

   * 실내/테라스, 케이지 필요, 대형견 가능, 시간 제한 등
4. 사진 업로드(선택)
5. 지도/리스트에서 조회 + 필터

### 3.2 예외/운영

* “동반 가능” 정보는 자주 바뀜 → 최신성 표시(작성일/최근 확인)
* 업주/마케팅 리뷰 악용 → 동일 IP/계정 도배 제한 + 신고 자동 블라인드

### 3.3 개발 포인트

* Place는 병원과 유사하게 “누적 페이지”가 필요(리뷰 집계/최근성)
* 태그는 단순 문자열 배열로 시작 → Phase에서 정규화(선택)

---

## 4) 산책로 공유(지도/코스)

### 4.1 정상 플로우

1. Local > 산책로 > **코스 등록**
2. 경로 입력

   * MVP: 시작점 + 경유지(단순)
   * 확장: 폴리라인(좌표 배열)
3. 태그 입력

   * 가로등/화장실/차량주의/결빙/주말 혼잡 등
4. 게시
5. 근처 사용자에게 추천(거리 기반)

### 4.2 예외/운영(개인정보 리스크)

* 집 위치 추정 방지:

  * 시작/종료 지점 자동 블러(반경 제거)
  * 정밀 좌표 저장 최소화(privacy 정책 준수)

### 4.3 데이터/상태

* `WalkRoute.coordinates JSON` (정밀도 정책 적용)
* 추천은 “푸시”보다 **피드/지도 카드 추천**부터 시작(MVP)

---

## 5) 산책 메이트 번개(모임)

### 5.1 정상 플로우

1. Local > 모임 > **모임 만들기**
2. 입력

   * 장소(공원/지점), 시간(즉시/예약), 인원수, 조건(대형견/사회성)
3. 참여(선착/승인)
4. 모임 진행
5. 종료 후 후기(선택) → 신뢰도 반영

### 5.2 예외/운영(악용이 빠름)

* 노쇼/분쟁:

  * 상태: `OPEN → FULL → CLOSED/CANCELLED`
  * 취소/노쇼 규칙(쿨다운, 신뢰도 하락)
* 개인 정보 노출:

  * 연락처 공개 금지(인앱 메시지)
* 위험 상황 신고:

  * 모임 신고 플로우를 단순화(원클릭 신고)

### 5.3 개발 포인트

* MVP는 “승인 없는 선착순”으로 단순화 가능
* 신뢰 지표(완료율/취소율)는 Phase2부터라도 스키마는 염두

---

## 6) 마켓(중고/대여/나눔) — 상태 머신이 핵심

### 6.1 등록(정상)

1. 마켓 > **등록**
2. 유형 선택: SELL / RENT / SHARE
3. 필수 입력

   * 카테고리, 상태, 사진, 위치(대략)
   * SELL: 가격
   * RENT: 대여료/보증금/기간
4. 게시(상태=AVAILABLE)

### 6.2 문의/거래 진행(정상)

* 구매/대여 문의 → 알림
* 판매자 상태 변경:

  * SELL/SHARE: `AVAILABLE → RESERVED → SOLD/CANCELLED`
  * RENT: `AVAILABLE → RESERVED → RENTED → RETURN_PENDING → AVAILABLE`
* 인수/반납 체크리스트(사진/확인)
* 후기(선택) → 신뢰 지표 축적

### 6.3 예외/운영(사기/분쟁)

* 외부 결제 유도/피싱 링크 탐지 → 자동 차단/제재
* 금지 품목/위험 품목 정책(리콜/위생/유통기한)
* 대여 분쟁:

  * 증빙 이미지 접근 통제(L3), 보존기간 설정(privacy 정책)

### 6.4 개발 포인트

* MVP는 “문의 + 상태 변경”까지만, 결제/정산은 Phase 이후
* 상태 변경은 반드시 서버에서 “허용 전이(transition)” 검증

---

## 7) 돌봄/심부름(마이크로 펫시팅) — “요청/수행/증빙”

### 7.1 요청 생성(정상)

1. 돌봄 > **요청 등록**
2. 유형 선택: 산책/급식/병원동행/긴급체크
3. 입력: 시간, 위치(대략), 요구사항, 보상(선택)
4. 주변 사용자에게 노출 + 알림(옵트인)

### 7.2 매칭/수행(정상)

1. 제공자 수락
2. 체크리스트 기반 수행(사진/타임라인)
3. 완료 처리
4. 후기/평점 → 신뢰도 반영

### 7.3 예외/운영(안전/사기)

* 연락처 교환 최소화(인앱 메시지)
* 악성 요청/허위:

  * 신규 계정 제한(가입 기간/동네 인증/활동 점수)
  * 취소/노쇼 누적 시 제재
* 증빙 데이터는 L3 취급(접근/보존기간)

### 7.4 개발 포인트

* MVP는 결제 없이도 가능하지만, “상태/로그/증빙”은 처음부터 설계해야 확장 가능
* 상태 예시: `OPEN → MATCHED → IN_PROGRESS → COMPLETED/CANCELLED`

---

## 8) 실종/유기 긴급 알림(Lost & Found) — 골든타임 플로우

### 8.1 신고 생성(정상)

1. Local > 실종/유기 > **긴급 신고**
2. 필수: 사진, 특징, 마지막 위치/시간, 연락 방법(노출 제한 옵션)
3. 게시 즉시 “긴급” 처리(우선 노출)
4. 반경/동네 기준 푸시(옵트인 + 근접 사용자)
5. 제보 등록(위치/시간/사진)
6. 종료 처리(찾음/보호소 인계) + 감사 메시지

### 8.2 예외/운영(허위/악용)

* 허위 신고 방지:

  * 생성 제한(가입 기간/동네 인증/이력)
  * 반복 허위는 즉시 제재
* 개인정보 보호:

  * 연락처 댓글 노출 금지(버튼/폼 방식)
  * 정밀 위치 저장 최소화(privacy 정책)
* 종료 후:

  * 종료 상태 명확화 + 더 이상 푸시/상단 노출 금지

### 8.3 데이터/상태

* `LostFoundAlert.status: ACTIVE → RESOLVED/CLOSED`
* 제보는 별도 엔티티로 분리 권장(향후 분석/검증 가능)

## 9) Global Q&A (지식 축적/SEO 플라이휠)

> 첨부 원문에 없지만, IA/전략상 핵심이므로 “최소 플로우”를 포함한다.

### 9.1 질문/답변(정상)

1. Global > Q&A > 질문 작성
2. 작성 전 **유사 질문 추천**(중복 방지)
3. 질문 게시
4. 답변 작성/업보트
5. 채택 → Q&A 품질 상승 → 검색 유입

### 9.2 운영/리스크

* 의료/훈련 조언 책임:

  * 면책/출처/주의 표기 가이드
  * 위험 주제는 자동 경고(“전문의 상담 권장”)

---

## 10) 공통 예외 플로우(운영 큐)

### 10.1 신고 처리(모든 콘텐츠 공통)

1. 사용자 신고(사유 선택 + 설명)
2. 서버: 레이트리밋/중복 신고 제한
3. 누적 기준 초과 시 자동 블라인드(soft hide)
4. 운영자 큐에서 검토:

   * 유지/삭제/제재/복구
5. 결과 기록(Audit Log)

### 10.2 계정 제재(공통)

* 단계: 경고 → 일시 정지 → 영구 정지
* 사유 템플릿(일관성) + 이의제기(Phase)

---

## 11) MVP 우선순위(개발 투입 기준)

### MVP에서 반드시 “끝까지” 되는 플로우

* 온보딩/동네 설정
* 병원 리뷰(템플릿) + 누적 페이지
* 장소 리뷰(템플릿)
* 산책로 등록(정밀도 정책 포함)
* 신고/자동 블라인드(최소 운영 기능)
* 레이트리밋(작성/검색/신고/업로드)

### Phase 2에서 확장

* 모임(노쇼/후기 포함)
* 마켓 상태 머신(대여 포함)
* 돌봄 상태/증빙/신뢰 지표
* L&F 반경 푸시 고도화

---

## 12) 리스크/주의(중요)

* “기능 먼저”로 Market/Care/L&F를 열면 악용이 먼저 온다 →
  **제한/증빙/제재/로그** 없으면 서비스가 망가진다.
* 산책로/실종은 위치 프라이버시가 가장 큰 폭탄 →
  정밀 좌표 저장/노출 정책을 코드로 강제(사람의 주의에 맡기지 않기).
