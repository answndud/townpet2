# Phase 2 (Local Network) 개발 순서/프로세스

목적
- Phase 2 기능을 실제 개발 순서대로 진행하며 기록한다
- 고위험 기능(마켓/돌봄/실종) 확장 시 정책 우선 원칙을 유지한다

전제
- Phase 1 핵심 루프와 운영 자동화 최소 버전 완료
- 신규 기능은 Local/Global 분리 원칙을 유지
- PLAN -> SCHEMA -> GENERATE -> REVIEW -> TEST -> DEPLOY 절차 준수

진행 표기 규칙
- [ ] TODO: 착수 전
- [~] DOING: 진행 중
- [x] DONE: 완료
- [!] BLOCKED: 외부 의존/정책 미확정으로 중단

Phase 2 목표
- 로컬 네트워크 효과 강화(번개/펫친소)
- 고위험 기능을 정책 기반으로 확장(마켓/실종/돌봄)
- 신뢰/운영 자동화 강화

세부 순서 (작성/진행 기록용)

1) 번개/펫친소 (Local)
- [ ] Meetup 스키마/상태 정의
- [ ] Zod 검증 + Service 정책(참여 제한/정원)
- [ ] 목록/상세/참여 플로우
- [ ] 알림/저장 루프 연결

2) 마켓 (Local)
- [ ] MarketListing 스키마/상태 머신 정교화
- [ ] 템플릿 필드(보증금/기간/상태/하자) 확정
- [ ] 신규 제한/링크 제한/신고 정책 강화
- [ ] 거래 상태 전이 및 체크리스트(인수/반납)

3) 실종/유기 (Local)
- [ ] LostFoundAlert 필수 필드 강화(위치/사진/연락)
- [ ] 반경 확산/알림 정책 정의
- [ ] 허위 신고 방지(신규 제한/검증)

4) 돌봄/심부름 수요 검증 (Local)
- [ ] 요청 템플릿(시간/위치/조건)
- [ ] 매칭/수락 상태 모델
- [ ] 증빙(사진/체크리스트/타임라인) 최소 버전
- [ ] 분쟁/취소 정책 초안

5) 운영/신뢰 강화
- [ ] 신고 누적 기준/시간창 재조정
- [ ] 관리자 큐 + 조치 이력 강화
- [ ] Rate limit 정책 재점검

6) 품질 보강
- [ ] Vitest 서비스 테스트 확장
- [ ] Playwright 핵심 플로우 보강
- [ ] 운영 로그/메트릭 연결(선택)

완료 기준(Definition of Done)
- 로컬 네트워크 기능이 정책과 함께 동작한다
- 마켓/실종/돌봄은 신규 제한 + 신고 흐름이 포함된다
- 템플릿 기반 입력으로 운영 리스크를 낮춘다
