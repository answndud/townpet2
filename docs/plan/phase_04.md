# Phase 4+ (Scale & Advanced) 개발 순서/프로세스

목적
- 실시간/푸시/결제 등 고도화 기능을 안정적으로 확장한다
- 운영/책임 리스크를 시스템으로 흡수한다

전제
- Phase 3 Global/Revenue 기능이 안정화됨
- 운영 정책(신고/제재/로그)이 충분히 성숙됨
- PLAN -> SCHEMA -> GENERATE -> REVIEW -> TEST -> DEPLOY 절차 준수

진행 표기 규칙
- [ ] TODO: 착수 전
- [~] DOING: 진행 중
- [x] DONE: 완료
- [!] BLOCKED: 외부 의존/정책 미확정으로 중단

Phase 4 목표
- 실시간 상호작용 강화
- 결제/멤버십 확장
- 운영 자동화/모니터링 고도화

세부 순서 (작성/진행 기록용)

1) 실시간 커뮤니케이션
- [ ] 채팅/실시간 알림 아키텍처 설계
- [ ] 기본 1:1/그룹 채팅 플로우
- [ ] 악용 방지(차단/신고/레이트리밋)

2) 푸시 알림
- [ ] FCM/웹푸시 아키텍처 정의
- [ ] 로컬 반경 알림 정책 정교화
- [ ] 사용자 알림 설정/해제 UX

3) 결제/정산
- [ ] 결제 프로바이더 선정/연동
- [ ] 환불/분쟁/취소 정책 확정
- [ ] 거래 로그/증빙 강화

4) 멤버십/광고 확장
- [ ] 멤버십 등급/혜택 정책 확정
- [ ] 광고 노출 기준/표기 고도화
- [ ] 수익/리텐션 KPI 추적

5) 운영 자동화 고도화
- [ ] 관리자 도구/모더레이션 개선
- [ ] 정책 위반 탐지 자동화(룰 기반)
- [ ] 감사 로그/데이터 보존 정책

6) 품질 보강
- [ ] 부하 테스트/성능 최적화
- [ ] 보안 점검/권한 재검토
- [ ] 장애 대응/복구 절차 문서화

완료 기준(Definition of Done)
- 실시간/푸시/결제가 정책과 함께 동작한다
- 운영 자동화가 성장 구간에서도 유지된다
- 보안/성능/모니터링이 제품 수준에 도달한다
