# privacy_data_handling.md — 개인정보/민감정보 처리 초안 (TownPet)

> 목적: TownPet 서비스에서 발생 가능한 **개인정보/민감정보/위치정보/콘텐츠(사진·영수증 등)**를 안전하게 처리하기 위한 내부 초안이다.
> 범위: 제품 기능(Local/Market/Care/Lost&Found/Global) 전반 + 운영/로그/분석/데이터 비즈니스(2차 활용).
> 원칙: **최소 수집, 목적 제한, 기본 비공개, 사용자 통제, 저장 최소화, 접근 최소화, 감사 가능성**.

---

## 0) 결론(짧게)

TownPet은 “동네 기반” 특성상 **위치·연락처·이미지**가 리스크의 대부분이다.
따라서 (1) 위치는 **정밀 좌표를 기본값으로 저장하지 않고**, (2) 연락은 **플랫폼 내 보호 채널**을 기본으로 하며, (3) 이미지는 **메타데이터 제거/자동 마스킹/보존기간**으로 통제한다.
데이터 비즈니스(통계/인사이트)는 **익명·집계·차등프라이버시 수준의 안전장치**를 전제로 한다.

---

## 1) 데이터 분류 체계

### 1.1 데이터 등급(Level)

* **L0: 공개 정보**
  예: 공개 게시물(동네 리뷰/산책로 글) 중 개인 식별 요소가 제거된 내용
* **L1: 일반 개인정보(PII)**
  예: 이메일, 닉네임(고유), 프로필 이미지, 기기 식별자(쿠키/세션), IP(로그)
* **L2: 민감/고위험 개인정보**
  예: 전화번호, 실명, 정확한 위치좌표, 집 주소, 결제 관련 식별정보(간접 포함)
* **L3: 특별 민감 정보/고위험 콘텐츠**
  예: 사진 속 얼굴/차량 번호판/집 내부, 영수증/진료내역(간접 의료), 분쟁 증빙, 신고/제재 기록

> L2~L3는 “기본 비공개 + 최소 저장 + 강한 접근통제”가 기본값.

### 1.2 기능별 주요 데이터(개요)

* **Auth/Account**: 이메일(OAuth), 닉네임, 프로필 이미지, 계정 식별자
* **Local**: 동네(행정동), 리뷰(병원비/대기), 산책로(좌표/경로)
* **Market**: 거래 게시물, 채팅/문의, 위치(대략), 증빙 이미지(상태/하자)
* **Care**: 요청 내용(시간/위치/조건), 수행 증빙(사진/체크리스트)
* **Lost&Found**: 사진/위치/연락, 반경 푸시 대상(근접 사용자)
* **Ops/Moderation**: 신고 내역, 제재 로그, IP/디바이스 일부(악용 방지)
* **Analytics**: 사용 이벤트(집계), 성능/오류 로그(Sentry 등)

## 2) 공통 원칙(필수 정책)

### 2.1 최소 수집 & 기본 비공개

* 기능에 꼭 필요한 데이터만 수집한다(“있으면 좋음”은 금지).
* 민감 데이터는 **기본값 비공개**이며, 사용자가 명시적으로 범위를 선택한다.

### 2.2 목적 제한 & 2차 이용 금지(기본)

* 수집 목적을 문서화하고, 다른 목적 이용은 원칙적으로 금지.
* 데이터 비즈니스(인사이트 제공/판매/제휴)는 **집계·익명화·옵트아웃** 전제로 별도 정책.

### 2.3 보존기간(저장 최소화)

* 데이터는 “영구 저장”이 아니라 **보존기간이 있는 자산**으로 취급한다.
* 민감/고위험(L2~L3)은 기본적으로 짧은 보존기간 + 자동 파기.

### 2.4 접근 최소화 & 감사 가능성

* 운영자/관리자 접근은 역할 기반(RBAC)으로 제한.
* L2~L3 접근은 **사유/티켓/감사 로그**를 남긴다.

### 2.5 사용자 권리

* 다운로드/정정/삭제(탈퇴 포함) 경로 제공
* 범위 설정(동네/온동네, 프로필 노출 범위)
* 위치/알림 등은 옵트인 + 언제든 변경 가능

## 3) 데이터 항목별 처리 가이드

## 3.1 위치정보(Location)

### 수집 원칙

* **정밀 좌표 저장은 기본 금지**. 필요한 경우에도 최소 정밀도로 저장한다.
* 기본값은 **행정동/그리드(예: 200~500m 단위)로 변환** 후 저장.

### 저장/노출 정책

* 게시물에 포함되는 위치:

  * **Local 리뷰/장소**: 장소 주소는 공개 가능하되, 사용자 개인 위치는 저장/노출 금지
  * **산책로 경로(좌표)**: 공개 시 “집 주변/개인 동선” 노출 위험 →

    * 기본 공개는 **축약(샘플링) + 시작/종료 지점 흐림(블러 처리)**
    * 특정 지점(집/회사 추정) 근처는 자동 마스킹(반경 N미터 제거)
  * **Lost&Found**: 마지막 목격 지점은 공개되지만, 신고자 집 주소/상세 위치는 비공개가 기본
* **반경 푸시**는 서버에서 계산하되, 사용자 좌표를 장기간 저장하지 않는다(가능하면 단기 세션/캐시).

### 보존기간

* 정밀 좌표(있다면): **최대 24~72시간**(긴급 기능/매칭 완료 후 파기)
* 행정동/그리드: 계정 설정 데이터로 유지 가능(사용자 변경/삭제 가능)

### 리스크/방어

* 스토킹/사생활 침해 방지:

  * “내 집 근처 산책로”의 시작점 노출 자동 차단
  * 특정 사용자 대상으로 위치 추적 시도 탐지(반복 조회/스크래핑 레이트리밋)

## 3.2 사진/동영상/첨부파일(영수증, 진료내역, 분쟁 증빙 포함)

### 위험 요소

* 얼굴, 아동, 집 내부, 차량 번호판, 위치 메타데이터(EXIF), 영수증의 결제정보/상호/시간/카드 단서

### 업로드 처리(필수)

* **EXIF/메타데이터 제거**: 업로드 시점에 자동 스트립
* **자동 마스킹 후보**(Phase):

  * 번호판/얼굴/문서(영수증) 영역 감지 후 블러(초기엔 수동 가이드 + 신고 기반도 가능)
* 파일 검증:

  * 확장자/마임/크기 제한
  * 악성 파일 방지(실행파일 차단)

### 표시/공유 정책

* 공개 게시물의 이미지:

  * 기본 공개는 가능하나, 개인 정보 포함 시 신고/자동 블라인드
* 증빙 이미지(분쟁/신고/돌봄 수행):

  * 기본은 **당사자+운영자만 접근**(L3)
  * 링크 공유 금지(서명 URL 단기 만료, 핫링크 방지)

### 보존기간

* 일반 이미지(게시물): 사용자가 삭제할 때까지(단, 정책 위반 시 블라인드/삭제)
* 분쟁/신고 증빙: **최대 90~180일** 보관 후 자동 파기(법적 이슈 발생 시 예외)
* Care 수행 증빙: **최대 30~90일** 보관(신뢰/이의제기 기간 이후 자동 파기)

---

## 3.3 연락처(전화번호, 이메일, 메신저 ID) & 커뮤니케이션

### 기본 정책

* 연락처는 가능한 한 **수집하지 않는다**.
* 거래/돌봄/실종은 “연락이 필요”하므로, **플랫폼 내 보호 채널**(in-app 메시지/폼)로 우회한다.

### 구체 규칙

* 게시물 본문/댓글에 전화번호/카톡ID 등 직접 노출:

  * 자동 탐지(정규식) → 경고/마스킹/숨김
* 거래/돌봄 매칭:

  * 초기에는 “문의 폼/메시지”로 연결하고, 전화번호 교환은 사용자 책임으로 두되
    **노출을 최소화**(버튼 클릭 시 1회 표시, 로그 기록, 스팸 신고 빠른 동선)
* Lost&Found:

  * 연락처는 “댓글 공개”가 아니라 **버튼 클릭으로 표시** 또는 **중계 폼** 제공
  * 악성 연락 방지(연락 제한, 차단/신고, 야간 제한 옵션)

### 보존기간

* 메시지/문의: 기본 1년(또는 사용자 설정)
* 스팸/악용 계정 신호: 최소한의 해시/서명 형태로 더 길게 보관 가능(악용 방지 목적)

---

## 3.4 결제/정산 데이터(Phase 3+)

> 결제는 리스크가 크므로 “저장 최소화”가 원칙이다.

* 카드정보 등 민감 결제정보는 **직접 저장 금지**(결제대행사 토큰화 사용)
* 영수증/정산 내역도 개인 식별이 가능한 수준이면 L2~L3로 취급
* 환불/분쟁 처리용 로그는 최소화 + 접근 통제 + 보존기간 명시

---

## 3.5 건강/의료 관련 데이터(간접 민감정보)

TownPet은 의료 서비스가 아니어도, 다음 데이터는 “간접 건강정보”가 될 수 있다.

* 병원비 상세(진료 항목), 처치/검사 언급, 알러지/질환 관련 질문/답변, 약 사진

정책:

* 사용자가 **자발적으로 게시**한 정보라도 민감정보로 간주 가능(L3)
* 노출 범위 옵션 제공(동네/전체/비공개 저장)
* 의료 조언은 “정보 제공” 수준으로 제한하고, 위험 신호는 병원/전문가 안내로 유도(법적 리스크 완화)

---

## 3.6 아동/가정/주거 노출(사진·게시물)

* 사진에 아동/집 내부/주소가 포함될 수 있음 → 신고/자동 감지 우선순위 상향
* 커뮤니티 가이드라인에 “집 위치 추정 가능 정보 금지” 명시

---

## 3.7 운영/모더레이션 데이터(신고, 제재, 블랙리스트)

### 원칙

* 신고/제재 기록은 L3(고위험)로 취급
* 공개 금지(명예훼손/2차 피해 방지)
* 운영자 접근은 최소화 + 감사 로그

### 보존기간

* 제재 이력: 악용 방지 목적의 최소 기간(예: 1~3년)
* 단, 저장 항목은 최소화(필요 시 해시/서명 기반)

---

## 3.8 로그/분석(Analytics, Error Tracking)

### 수집 원칙

* 이벤트는 기본적으로 **익명/집계**로 설계
* IP/UA 등은 보안/운영에 필요할 경우만 제한적으로 보관

### 설정

* Sentry 등 오류 로그:

  * PII 자동 마스킹 설정(가능한 범위 내)
  * request body/헤더 중 민감값 수집 금지
* 제품 분석(예: Vercel Analytics/Plausible):

  * 가능한 한 쿠키 최소/비식별 방식 선택
  * 내부적으로 “사용자 행동 프로파일링”은 기본 금지(필요 시 옵트인)

---

## 4) 데이터 비즈니스(2차 활용) — 원칙/금지/허용

### 4.1 금지(기본)

* 개인 단위 데이터 판매/제공(직접·간접 식별 포함)
* 특정 동네/소수 집단을 식별할 수 있는 미세 집계(재식별 위험)
* 외부 제휴를 위한 “원본 데이터 제공”

### 4.2 허용(조건부)

* **집계 통계**: 예) 동네별 병원 대기 “중앙값 범위”, 항목별 비용 “구간 분포”
* **k-익명성 기준**: 표본이 충분할 때만 공개(예: k≥20 등 내부 기준)
* **차등 프라이버시(선택)**: 공개 통계에 노이즈 추가(Phase)
* **옵트아웃**: 사용자가 내 데이터가 통계에 포함되는지 거부 가능(기본 포함/기본 제외 중 정책 선택)

### 4.3 제품에서의 표기

* “이 정보는 사용자 제보 기반이며, 표본 수가 적을 수 있음” 같은 한계 표기
* 표본 수가 적을 때는 범위만 제공하거나 비노출

---

## 5) 보안/접근 통제(구현 지침)

### 5.1 권한/접근

* 역할 기반 접근(RBAC): USER / MODERATOR / ADMIN
* L2~L3 데이터 접근은 “업무 사유” + 감사 로그 필수

### 5.2 저장소/전송

* 전송: HTTPS 기본
* 저장: DB 암호화(매니지드 기능 활용) + 민감 필드(전화/주소/정밀 위치)는 추가 암호화 고려
* 파일: private bucket 기본 + 서명 URL 단기 만료

### 5.3 키/시크릿

* 키는 코드 저장 금지(.env) + 배포 환경 비밀 저장소
* 키 회전 정책(주기/사고 시 즉시)

### 5.4 취약점 대응

* 업로드/다운로드 URL 무차별 대입 방지(랜덤 키 + 만료 + 레이트리밋)
* 스크래핑 방지: 레이트리밋/봇 탐지/의심 행위 차단

---

## 6) 사용자 UI/UX 가이드(프라이버시를 기본값으로)

* 위치 공유: “정확한 위치”가 아니라 “대략 위치” 기본값
* 게시물 작성 시 민감 정보 경고(연락처/주소/차량번호 등 감지)
* 범위 선택: Local/Global/비공개 저장(북마크형)
* 탈퇴 시: 삭제 범위 안내(댓글/게시물/집계 통계 포함 여부)
* 실종 글: 연락처 공개 방식(버튼 클릭/폼 중계) 선택 제공

| 이메일(OAuth) |    L1 |    계정 유지 | 탈퇴 시 삭제(법적 예외 제외) |
| 전화번호       |    L2 | 가능하면 미수집 | 불가피 시 암호화 + 최소 보존 |
| 정밀 위치좌표    |    L2 |    저장 지양 | 필요 시 24~72h 후 파기  |
| 행정동/그리드    | L1~L2 |    계정 설정 | 사용자가 변경/삭제        |
| 게시물 이미지    | L1~L3 |    게시 유지 | 메타데이터 제거, 신고 시 제한 |
| 분쟁/신고 증빙   |    L3 |  90~180일 | 이의제기 기간 후 자동 파기   |
| Care 수행 증빙 |    L3 |   30~90일 | 분쟁 기간 후 자동 파기     |
| 신고/제재 기록   |    L3 |     1~3년 | 최소 정보만 보존         |
| 분석 이벤트     | L0~L1 |   13개월 등 | 비식별/집계 우선         |

> 숫자는 초안이며, 실제는 법/정책/운영 리스크에 맞춰 확정.

## 8) 추가로 고려해야 할 데이터(사용자 언급 외 핵심)

당신이 언급한 “위치/영수증·사진/연락처/데이터 비즈니스” 외에, 실제로 자주 터지는 항목들:

* **신고/제재 데이터(운영 기록)**: 내부에서도 접근 제한이 강해야 함
* **돌봄 수행 증빙(사진/체크리스트)**: 사고/분쟁의 핵심 증거 → 보존기간/접근통제가 중요
* **메시지/문의(채팅 유사)**: 개인정보가 섞이기 쉬움 → 자동 탐지/마스킹 필요
* **로그/분석(Sentry/Analytics)**: 의도치 않게 PII가 섞이는 대표 영역
* **사용자 생성 콘텐츠(UGC) 재식별 위험**: “동네 + 시간 + 사진” 조합으로 개인이 특정될 수 있음
