# TownPet 내부용 개발 Overview (최신 운영 기준)

기준일: 2026-02-26
대상: TownPet 코드베이스에 바로 투입되는 개발자/에이전트
참조: `PLAN.md`, `PROGRESS.md`, `FORcodex.md`, `app/prisma/schema.prisma`

---

## 1) 현재 제품 상태 요약

- TownPet은 `LOCAL`/`GLOBAL`을 섞지 않고 분리하는 반려 커뮤니티다.
- 현재 기본 사용자 흐름은 `Global-first`다. 대표 동네가 없어도 피드/검색/글쓰기를 시작할 수 있다.
- 커뮤니티 구조는 `커뮤니티 보드(COMMUNITY)` + `공용 보드(COMMON)` 이원화가 적용되어 있다.
- 비회원 작성(게스트 작성자)은 허용되지만, 강한 정책(레이트리밋/링크 제한/고위험 카테고리 제한)을 기본 적용한다.

---

## 2) 제품 원칙 (변경 금지)

1. Local vs Global 분리는 필터가 아니라 운영 규칙이다.
2. 고위험 기능은 정책이 기능보다 먼저다.
3. 입력은 템플릿/검증(Zod)으로 구조화하고, 자유서술은 보조로 다룬다.
4. 정책은 UI가 아니라 서비스 레이어에서 집행한다.
5. 운영 자동화(신고/제재/레이트리밋/로그)가 없는 기능은 출시하지 않는다.

---

## 3) 기술 스택 (현재 코드 기준)

- Next.js App Router (`next@16.x`)
- React (`react@19.x`)
- TypeScript strict
- Prisma + PostgreSQL
- Zod validation
- NextAuth v5 (Credentials + Kakao/Naver + dev social)
- Upstash Redis compatible rate limiting (메모리 fallback)
- 서버 검증 업로드 파이프라인 + `/media/uploads/...` canonical proxy + Vercel Blob/local storage
- Sentry 연동 가능한 에러 모니터링

주의:
- 과거 문서의 Next.js 14 기준 설명은 참고용이며, 현재 구현 기준은 app 패키지 버전을 따른다.

---

## 4) 아키텍처 핵심

### 4.1 레이어

1. UI: `app/src/app`, `app/src/components`
2. 경계: Route Handlers / Server Actions
3. 정책/비즈니스: `app/src/server/services`
4. 조회 최적화: `app/src/server/queries`
5. 데이터 모델: `app/prisma/schema.prisma`

### 4.2 구현 순서 표준

`Prisma -> Zod -> Service -> Action/Route -> UI -> Tests`

---

## 5) 현재 도메인 구조 (핵심)

### 5.1 게시글 보드 구조

- `Post.boardScope = COMMUNITY | COMMON`
- `Post.communityId` (커뮤니티 글)
- `Post.commonBoardType = HOSPITAL | LOST_FOUND | MARKET` (공용 보드 글)
- `Post.animalTags[]` (공용 보드 탐색 태그)

### 5.2 작성자 구조

- 회원: `User`
- 비회원: `GuestAuthor`
- 게시글/댓글에서 guest identity를 별도 모델로 추적하고 legacy hash와 호환 경로를 유지한다.

### 5.3 범위 구조

- `Post.scope = LOCAL | GLOBAL`
- 기본 피드/API는 Global-first로 동작하며, LOCAL은 대표 동네 설정 시 활성화된다.

---

## 6) 필수 정책 가드레일

### 6.1 안전/어뷰징

- 로그인/조회/작성/신고 경로 레이트리밋
- 신규 계정 제한(고위험 카테고리)
- 링크/연락처 제한
- 신고 누적 기반 자동 숨김
- 금칙어 및 단계적 제재

### 6.2 인증/보안

- 인증 응답 enumeration 완화
- 로그인 락아웃(account + ip)
- 비밀번호 강정책 + 취약 비밀번호 차단
- client IP 파싱을 trusted proxy hops 기준으로 정책화
- health endpoint 공개/내부 진단 분리

## 10) 지금의 우선순위

- 성장 기능보다 안전/운영 자동화 완성도를 우선한다.
- 커뮤니티 밀도는 좁은 범위에서 먼저 만들고, 카테고리 확장은 지표 기준으로 단계 오픈한다.
- 제품 "완성"의 기준은 UI 노출이 아니라 정책/운영/테스트까지 통과한 상태다.
