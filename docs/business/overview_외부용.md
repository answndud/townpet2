# TownPet Overview (외부용)

* **문서 목적**: 투자자/파트너/외부 이해관계자가 TownPet의 “무엇을/왜/어떻게”를 빠르게 파악하도록 하는 단일 개요 문서
* **기준 문서**: 비즈니스 오버뷰, 마스터 Overview, 프로젝트 기능 소개   
* **마지막 업데이트(기술 명세 기준)**: 2026-01-21

---

## 1) 한 줄 정의

**TownPet은 ‘동네 기반(Local) 반려생활 인프라’와 ‘지역 무관(Global) 지식 축적’을 분리 운영하여, 로컬 재방문(리텐션)과 검색 유입(SEO)을 동시에 확보하는 반려동물 커뮤니티 플랫폼**이다.  

---

## 2) 해결하려는 문제

### Local(동네) 문제

* 병원 비용/대기/서비스, 동반 가능 장소, 산책로 안전 같은 **생활 밀착 정보가 분산**되어 있고 **최신성·신뢰성**이 낮다. 
* 마켓/돌봄/구인은 **스팸·사기·분쟁**이 빠르게 유입되는 영역이라 “기능만 있는 커뮤니티”는 유지가 어렵다.  

### Global(지식) 문제

* 견종/사료/행동/건강 정보는 축적될수록 가치가 커지지만, 로컬 피드와 섞이면 **피드 혼탁**으로 즉시성 가치가 희석된다.  

---

## 3) 핵심 해법(제품 원칙)

1. **Local vs Global 분리(탭/검색/알림/정책까지 분리)**

* Local은 거리·최신성·즉시성이 본질 → **리텐션/재방문이 성공 기준**
* Global은 축적·정합성·전문성이 본질 → **검색성/전문성이 성공 기준** 

2. **템플릿 기반 UGC(구조화 입력)**

* 병원 리뷰(방문일/진료 항목/비용/대기), 대여(보증금/기간/체크리스트)처럼 **자유서술 이전에 구조화 필드**를 받아 데이터 품질을 고정한다.   

3. **신뢰/운영 자동화(코드로 정책을 모델링)**

* 신고 누적 기반 자동 블라인드(HIDDEN), 신규 계정 제한, 링크/연락처 제한, 레이트리밋, 관리자 큐 및 조치 이력(Audit Log) 등 **운영 비용을 시스템화**한다.   

---

## 4) 제품 구성(핵심 기능)

### 4.1 Local (동네 기반)

* **동네 인프라 비교**: 병원 템플릿 리뷰(비용/대기/항목), 동반 가능 장소(생활형 장소 포함) 
* **산책로 공유(지도)**: 코스/난이도/가로등/주차/화장실/안전 태그 기반  
* **번개/펫친소**: 동네 네트워크 효과를 만드는 모임/프로필 탐색 
* **마켓(중고/대여/나눔)**: 대여는 보증금/기간/인수·반납 체크리스트로 분쟁 최소화  
* **심부름/긴급 돌봄(마이크로 펫시팅)**: 초기에는 결제 없이도 요청/매칭/후기로 수요 검증 가능 
* **실종/유기 알림**: 위치 기반 푸시로 인근 주민에게 확산(골든타임) 

### 4.2 Global (지역 무관)

* **품종 토픽**: 견종/묘종 특화 지식 커뮤니티 
* **Q&A(채택/정렬/중복 억제)**: 지식 자산화 구조 
* **사료/용품 DB**: 성분/알러지 기반 필터 + 리뷰 축적 
* **전문가 콘텐츠**: 자격/면책/검수 정책을 전제로 신뢰 확보 

---

## 5) 비즈니스 모델(수익화 설계)

### Local BM

* **지역 타겟 광고**: 상단 노출/쿠폰/프로필 강화(운영 비용 낮은 순서로 시작) 
* **중개 수수료**: 펫시팅/도그워커 매칭·예약(결제/분쟁이 붙는 순간 운영 복잡도 급증 → 후반 Phase 권장) 
* **동네 공동구매**: 모집/마감만으로도 초기 검증 가능(결제는 후순위) 

### Global BM

* **어필리에이트**: “광고 추천”이 아니라 “정보 기반 필터 추천”으로 신뢰 유지 
* **프리미엄 멤버십**: 광고 제거/전문가 상담권/특가(의미 있는 재방문이 선행 조건) 
* **데이터 비즈니스(장기)**: 동의/가명/집계 기반으로만 가능(Phase 후반) 

---

## 6) 성공 지표(KPI) — “성공의 정의”를 고정

* **Local**: 7일 리텐션(동네 코호트), 템플릿 완성률(비용/대기/조건 입력), 저장/검색 사용률, (실종) 알림→제보 전환 및 제보 시간  
* **거래/돌봄**: 문의율, 성사율, 분쟁/신고율, 완료율/노쇼율 
* **Global**: 질문당 답변 수, 채택률, 자연 검색 유입, 검색→체류 시간, DB 재방문률 

---

## 7) 운영/신뢰(생존 조건)

* **신규 계정/활동 점수 기반 기능 제한**(마켓/돌봄/긴급 신고 등 고위험 기능 보호) 
* **신고 누적 기반 자동 블라인드 + 관리자 큐** 
* **링크/연락처 노출 제한 + 레이트리밋**으로 스팸 억제  
* **의료/전문 영역 리스크 관리**: 경험 공유 vs 진단/처방 분리, 면책/표기/검수 정책 고정 

---

## 8) 로드맵(Phase 기반)

* **Phase 1 (MVP Core)**: 인증/프로필/동네 설정 → 게시물 코어 → 병원·장소 리뷰 → 산책로(지도) → 신고/운영
* **Phase 2 (로컬 네트워크)**: 번개/펫친소, 마켓/대여/나눔, 실종/유기(긴급 알림), 돌봄 수요 검증
* **Phase 3 (Global & 수익)**: Q&A/토픽/DB/어필리에이트 → 멤버십 실험
* **Phase 4+ (고도화)**: 실시간/푸시/결제/멤버십 확장

(문서 세트 내 로드맵·BM 실행 순서와 정합) 

---

## 9) 기술 개요(외부용 요약)

* **현재 기술 명세의 방향**: TypeScript 단일 언어 기반 **풀스택(Next.js) + Prisma + PostgreSQL + 매니지드 인프라**로 1인+AI 협업 개발을 최적화
* **품질/운영 포인트**: 타입 안정성 + 입력 검증(스키마) + 레이트리밋 + 신고/블라인드 자동화 + 모니터링 기반 운영

> 참고: 업로드된 “프로젝트 소개(이력서용)” 문서에는 Spring Boot 중심의 구현 예시가 포함되어 있으나, 본 SPEC.md는 TypeScript 단일 스택으로 재정의되어 있다.

## 10) 핵심 리스크와 대응

* **스팸/업자/사기(마켓/구인/돌봄)**: “기본값이 스팸” 전제 → 제한/레이트리밋/자동 블라인드/신고 큐 
* **실종/유기 기능 악용(허위 긴급 신고)**: 생성 조건 강화 + 빠른 운영자 검토 큐 + 반복 악용 제재 
* **의료/전문 조언 책임**: 표준 면책/표기/검수 규정으로 “진단/처방처럼 보이는 콘텐츠” 차단 
* **결제 도입 시 운영비 폭증**: 결제=CS/환불/분쟁 시스템 → 정책/증빙/로그 체계를 먼저 갖춘 뒤 후반 Phase에서 도입
