# TownPet GUIDE (다른 터미널 복붙 즉시 실행)

이 문서는 Codex 터미널이 아니라 본인 터미널(iTerm/Terminal)에서 바로 실행되게 작성했습니다.
명령어는 모두 절대경로 기준이라, 현재 위치와 무관하게 복붙 가능합니다.

## 1. 한 번에 준비하기 (DB + 스키마 + 더미데이터)

아래 블록을 터미널에 그대로 붙여넣으세요.

```bash
export TOWNPET_HOME="/Users/alex/project/townpet"

open -a Docker
cd "$TOWNPET_HOME" && docker compose up -d

cd "$TOWNPET_HOME/app" && pnpm install
cd "$TOWNPET_HOME/app" && pnpm db:migrate

cd "$TOWNPET_HOME/app" && SEED_DEFAULT_PASSWORD=dev-password-1234 pnpm db:seed
cd "$TOWNPET_HOME/app" && SEED_DEFAULT_PASSWORD=dev-password-1234 pnpm db:seed:users
cd "$TOWNPET_HOME/app" && pnpm db:seed:reports
```

정상 기준:
- `docker compose ps`에서 `postgres`가 `Up`
- `seed-users.ts` 실행 후 `Seed users ready.` 출력

기본 원칙:
- 로컬 스키마 적용 기본 경로는 `pnpm db:migrate`(`prisma migrate dev`)입니다.
- `pnpm db:push`는 disposable local DB를 빠르게 맞추거나 임시 복구가 필요할 때만 사용하고, 일반적인 개발/운영 절차의 기준선으로 사용하지 않습니다.

## 2. 개발 서버 실행 (본인 새 터미널)

권장 실행 방법은 `pnpm dev` 입니다.  
현재 기본 `dev` 스크립트는 Turbopack panic(`Next.js package not found`) 회피를 위해 `webpack` 모드로 고정되어 있습니다. Turbopack이 꼭 필요할 때만 `pnpm dev:turbo`를 사용하세요.
이 프로젝트는 `dev` 스크립트에 `prisma generate`가 포함되어 있어 Prisma Client 누락 오류를 예방합니다.

```bash
cd /Users/alex/project/townpet/app && pnpm dev
```

포트를 고정해서 실행해야 할 때만 아래 대안을 사용하세요.

```bash
cd /Users/alex/project/townpet/app && ./node_modules/.bin/prisma generate && ./node_modules/.bin/next dev --webpack -p 3000
```

접속 주소:
- `http://localhost:3000`

## 4-1-3. 배포 전 auth email readiness preflight (복붙)

`citext` 전환 전후 운영 DB가 case-insensitive email 기준으로 안전한지 확인합니다.

```bash
cd /Users/alex/project/townpet/app && pnpm ops:check:auth-email-readiness
```

정상 기준:
- `summary: pass=5, warn=0, fail=0`
- duplicate / invalid normalized row가 없어야 함

실행 시 확인 항목:
- `User.email`의 `trim+lowercase` 기준 duplicate
- `User.email`의 normalization drift
- `VerificationToken.identifier`의 normalization drift

배포 자동화 기준:
- production `build:vercel`은 `ops:check:security-env:strict` 다음에 `prisma generate`, 그 다음 auth email readiness preflight를 자동 실행
- preview는 기본 skip이며 `DEPLOY_AUTH_EMAIL_PREFLIGHT_STRICT=1`로 opt-in 가능
- 긴급 우회가 필요하면 `DEPLOY_AUTH_EMAIL_PREFLIGHT_SKIP=1`로 강제 skip 가능

## 4-1-1. 운영 지연 스냅샷 수집 (복붙)

배포 API(`posts/suggestions/breed/search-log`)의 지연 샘플을 수집하고 p50/p95 요약을 생성합니다.

```bash

```

기본 출력:

- summary md: `/tmp/townpet_latency_snapshot_<timestamp>.tsv.summary.md`

옵션:
- `OPS_PERF_GET_SAMPLES` (기본 30)
- `OPS_PERF_POST_SAMPLES` (기본 20)
- `OPS_PERF_PAUSE_MS` (기본 200)
- `OPS_PERF_WARMUP_SAMPLES_PER_ENDPOINT` (기본 1, threshold 평가에서 제외)

판정 기준:
- `Threshold Evaluation`은 `Steady-state Samples`(warm-up 제외)를 기준으로 PASS/FAIL을 계산
- warm-up 구간은 summary의 `Warm-up Samples` 섹션에서 별도 관찰

GitHub Actions 자동 수집:
- 워크플로우: `.github/workflows/ops-latency-snapshots.yml`
- 스케줄: 하루 3회(UTC 00:10/08:10/16:10)

## 4-7. 알림 보관/90일 retention 정리 작업 (복붙)

알림은 읽음 처리 시 `isRead/readAt`만 갱신되어 inbox에 남고, `보관` 시 `archivedAt`으로 숨김 처리됩니다.
보관된 알림은 기본 90일 동안 유지되며, 그 이후 정리 작업에서 영구 삭제됩니다.

수동 실행:

```bash
cd /Users/alex/project/townpet/app && pnpm db:cleanup:notifications
```

참고:
- GitHub Actions `notification-cleanup` 워크플로우가 매일 실행됩니다.
- 워크플로우 기본 보존 기간은 `NOTIFICATION_RETENTION_DAYS=90`입니다.
- 운영 DB에 `archivedAt` 컬럼이 없는 경우 워크플로우에서 repair SQL을 먼저 적용합니다.
- 지원/운영 확인이 필요하면 아래 쿼리로 특정 사용자의 숨김 이력을 바로 확인할 수 있습니다.

```sql
SELECT "id", "title", "isRead", "readAt", "archivedAt", "createdAt"
FROM "Notification"
WHERE "userId" = 'USER_ID'
  AND "archivedAt" IS NOT NULL
ORDER BY "archivedAt" DESC
LIMIT 30;
```

입력 위치:

실제 발급/입력 방식(고정 메모):
- `DATABASE_URL`
  - `neon.tech`에서 발급
  - Connection string을 `Prisma` 형식으로 선택
  - 복사 후 Vercel `DATABASE_URL`에 그대로 붙여넣기

- `NEXTAUTH_URL`

주의:
- `DATABASE_URL` 오타 방지 (`DATABASE_URE` 아님)
- 환경변수 변경 후 반드시 Redeploy

참고:
- production 배포는 strict security env preflight와 runtime env 검증 때문에 `RESEND_API_KEY`, `BLOB_READ_WRITE_TOKEN`, `UPSTASH_REDIS_REST_URL/TOKEN` 누락 시 build 초반에 실패합니다.

### 10-1) Vercel 프로젝트 설정

위치:

핵심:

### 10-1-A) Prisma P3005 baseline 복구 절차(운영 DB 1회)

증상:

원인:

1. **운영 DB 백업 확보**

export DATABASE_URL='운영 DATABASE_URL'

  20260123025834_init
  20260123042307_add_post_categories
  20260123080655_add_report_audit

### 10-1-B) Prisma P3009 failed migration 복구 절차(로컬/운영 공통)

증상:
- `prisma migrate deploy` 또는 `prisma migrate status`에서 `P3009` failed migration 경고가 표시
- `_prisma_migrations`에 `finished_at IS NULL AND rolled_back_at IS NULL` 상태의 실패 이력이 남아 있음

원인:
- 부분 적용된 migration이 중간에 실패했거나, 기존 데이터가 새 제약조건과 충돌

1. **실패 migration 이름 확인**

2. **원인 데이터 정리**

```sql
SELECT id, "targetType", "targetId", "createdAt"
FROM "Report"
WHERE "targetType" <> 'POST'
ORDER BY "createdAt" DESC;

```

- GitHub Secrets 값은 저장 후 원문 재조회가 불가하므로 원본은 별도 비밀번호 관리자에 보관
- Vercel env와 GitHub secrets는 별개 저장소이므로 둘 다 입력해야 워크플로우와 런타임이 함께 동작

### 10-6) 워크플로우 실행 기준

- `oauth-real-e2e`: OAuth secret 검증 + 공급자 리다이렉트 스모크 + 앱 온보딩/피드 회귀(`social-dev`)
- `ops-smoke-checks`:
  - `target_base_url=https://<운영도메인>`
  - Sentry 검증이 필요 없으면 `Run Sentry ingestion verification` 해제해도 health 체크는 통과 가능
  - `pg_trgm` 설치 확인이 필요하면 `Run pg_trgm extension verification` 활성화(`HEALTH_INTERNAL_TOKEN` 필요)

## 11. 회원/게시글/이미지 정보는 어디서 보고 관리하나요?

핵심 정리:
- Vercel은 배포/로그/환경변수 관리 도구
- 회원/게시글 같은 데이터 관리는 앱 관리자 화면 + DB 콘솔(Neon)에서 수행

### 11-1) Vercel에서 가능한 것

- 배포 성공/실패, 빌드 로그 확인
- Function/Runtime 에러 로그 확인
- 도메인/환경변수 관리

### 11-2) 앱 관리자 화면에서 가능한 것

- 신고/제재 관리: `/admin/reports`
- 인증 감사 로그: `/admin/auth-audits`
- 정책/개인화 튜닝 관리: `/admin/policies`
- 개인화 지표 확인: `/admin/personalization`
- 품종 사전 관리: `/admin/breeds`

### 11-2-A) 인증 감사 로그 보존 기준

- 기본 retention:
  - 인증 감사 로그는 기본 `180일` 유지
- 정리 기준:
  - GitHub Actions `auth-audit-cleanup`이 하루 1회 실행
  - 삭제 대상은 `createdAt < now() - 180 days`인 `AuthAuditLog`
- 수동 실행:

```bash
cd /Users/alex/project/townpet/app
AUTH_AUDIT_RETENTION_DAYS=180 pnpm db:cleanup:auth-audits
```

### 11-2-B) 운영자가 같이 알아야 하는 사용자 화면

- 내가 작성한 글 확인/지원 응대: `/my-posts`
- 북마크한 글 확인/지원 응대: `/bookmarks` (`/saved`는 호환 리다이렉트)
- 피드 개인화 설명 확인: `/feed`
- 품종 라운지/세그먼트 노출 확인: `/lounges/breeds/[breedCode]`
- 프로필 반려동물/세그먼트 편집: `/profile`

### 11-3) DB(Neon)에서 가능한 것

- 회원/게시글/이미지 원본 데이터 조회
- 필요 시 SQL로 수정/삭제 가능(운영 주의)

예시 조회 쿼리(Neon SQL Editor):

```sql
SELECT id, email, role, "createdAt"
FROM "User"
ORDER BY "createdAt" DESC
LIMIT 50;
```

```sql
SELECT p.id, p.title, p."authorId", p."createdAt", p.status
FROM "Post" p
ORDER BY p."createdAt" DESC
LIMIT 50;
```

```sql
SELECT pi."postId", pi.url, pi."createdAt"
FROM "PostImage" pi
ORDER BY pi."createdAt" DESC
LIMIT 50;
```

마이그레이션 적용 여부 확인:

```sql
SELECT migration_name, started_at, finished_at, rolled_back_at
FROM "_prisma_migrations"
ORDER BY started_at DESC;
```

```sql
SELECT migration_name, started_at, finished_at, rolled_back_at
FROM "_prisma_migrations"
WHERE migration_name = '20260307030000_expand_report_reason_triage';
```

특정 migration의 적용 여부를 boolean으로 보고 싶을 때:

```sql
SELECT EXISTS (
  SELECT 1
  FROM "_prisma_migrations"
  WHERE migration_name = '20260307030000_expand_report_reason_triage'
    AND finished_at IS NOT NULL
    AND rolled_back_at IS NULL
) AS migration_applied;
```

enum 값 검증 예시(`ReportReason`):

```sql
SELECT array_agg(e.enumlabel ORDER BY e.enumsortorder) AS report_reason_values
FROM pg_enum e
JOIN pg_type t ON e.enumtypid = t.oid
WHERE t.typname = 'ReportReason';
```

중요:
- 비밀번호는 평문 조회 불가(해시만 저장)
- 비밀번호 확인 대신 재설정 플로우로만 처리

### 11-4) 이미지 업로드 저장 위치

- 현재 구현은 서버 파일시스템(`public/uploads`) 기반
- 로컬 개발: `app/public/uploads`
- Vercel 운영: `BLOB_READ_WRITE_TOKEN` 설정 시 Vercel Blob(`https://*.public.blob.vercel-storage.com`)에 저장
- 앱이 참조하는 canonical 경로는 `/media/uploads/...`이며, local/blob source는 서버 proxy route가 해석
