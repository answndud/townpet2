# SearchTermStat 전환 가이드

목표: 검색어 집계를 `SiteSetting(popular_search_terms_v1)` 구형 경로에서 `SearchTermStat` 단일 경로로 전환한다.

## 전환 대상
- 읽기: `getPopularSearchTerms`
- 쓰기: `recordSearchTerm`
- 저장소: `SearchTermStat` 테이블

## 개인정보/민감 검색어 처리
- `POST /api/search/log`는 응답 전에 `recordSearchTerm`을 await한다.
- 이메일, 전화번호, 오픈카카오/메신저 링크, 카카오톡 ID 패턴은 통계 저장에서 제외한다.
- 민감 검색어는 API에서 `recorded=false`, `skippedReason=SENSITIVE_TERM`으로 응답되며 인기 검색어 읽기에서도 한 번 더 필터링된다.

## 운영 적용 순서
1. 앱 배포 전 `SearchTermStat`가 포함된 마이그레이션 적용
   - `app/prisma/migrations/20260219184000_add_search_term_stats/migration.sql`
2. 앱 배포 후 검색어 로그 API 정상 동작 확인
   - 일반 검색어로 `POST /api/search/log` 호출 후 `SearchTermStat.count` 증가 확인
   - 이메일/전화번호 형태 검색어는 `recorded=false`로 응답하고 `SearchTermStat`에 저장되지 않는지 확인
3. 인기 검색어 노출 확인
   - 검색 페이지 추천어 섹션이 정상 렌더링되는지 확인

## 롤백 전략
- 코드 롤백 시에도 `SearchTermStat` 테이블은 유지해도 무방하다.
- 데이터 손실 위험이 없으므로 우선 앱 코드만 롤백하고 DB 스키마는 유지한다.

## 구형 데이터 정리
`SiteSetting`의 `popular_search_terms_v1` 키는 더 이상 사용되지 않는다.
운영 반영 시 아래 스크립트 기반으로 정리한다.

```bash
cd /Users/alex/project/townpet2/app && pnpm db:cleanup:legacy-search-setting
```

실제 삭제 실행:

```bash
cd /Users/alex/project/townpet2/app && pnpm db:cleanup:legacy-search-setting -- --apply
```

## retention 정리
`SearchTermStat`는 raw event log가 아니라 집계 테이블이지만, 오래된 term이 계속 남지 않도록 `updatedAt` 기준 retention cleanup을 수행한다.

로컬/수동 실행:

```bash
cd /Users/alex/project/townpet2/app && SEARCH_TERM_RETENTION_DAYS=90 pnpm db:cleanup:search-terms
```

GitHub Actions:
- 워크플로우: `.github/workflows/search-term-cleanup.yml`
- 기본 보존 기간: `90일`
- 필요 secrets: `DATABASE_URL`
