# OpenCode Agent-Only 운영 가이드 (TownPet)

작성일: 2026-02-25
목표: `commands` 없이 `@agent` 직접 호출만으로 운영한다.

## 1) 운영 원칙
- 커스텀 커맨드는 사용하지 않는다.
- 세션 시작/실행/검증/기록은 모두 `@agent` 호출로 처리한다.
- 1인 운영 기준으로 단순성과 재현성을 우선한다.
- 모든 작업은 `요청 -> 실행 -> 검증 -> 기록` 4단계를 남긴다.

## 1.1 근거 (공식 문서 + 실사용 패턴)
- OpenCode Agents 문서: subagent는 `@mention` 직접 호출 가능, primary가 Task로 위임도 가능.
- OpenCode Commands 문서: commands는 반복 프롬프트 단축용 선택 기능(필수 아님).
- 운영 결론: TownPet는 agent-only가 맞다. 이유는 다음 3가지.
  - 명시적 제어: 어떤 에이전트가 동작하는지 즉시 확인 가능
  - 유지보수 단순화: 명령 템플릿 중복 제거
  - 회귀 감소: 커맨드 라우팅 누락/오타 리스크 제거

## 2) 현재 에이전트 구성
- `@orchestrator`: 요청 분류/라우팅 전용
- `@delivery-engineer`: 기능/버그/성능/리팩터 구현
- `@safety-verifier`: QA/정책/릴리스 게이트 검증 (read-only)
- `@growth-operator`: SNS/커뮤니티 성장 실행안
- `@local-error-fixer`: 로컬 에러 로그 즉시 수정
- `@plan-coordinator`: `PLAN.md`/`PROGRESS.md` 동기화

## 2.1 에이전트 선택 규칙 (즉시 판정)
- 코드/스키마를 바꿔야 한다 -> `@delivery-engineer`
- 에러 로그가 있다 -> `@local-error-fixer`
- 변경 완료 후 머지 가능 여부 판단 -> `@safety-verifier`
- 온보딩/유입/SNS/커뮤니티 확장 -> `@growth-operator`
- 요청이 섞여 있고 시작점이 불명확 -> `@orchestrator`
- PLAN/PROGRESS 상태 반영 -> `@plan-coordinator`

## 3) 기본 실행 플로우
1) 계획 동기화
   - `@plan-coordinator 이번 세션 목표를 PLAN/PROGRESS에 등록하고 in_progress 1개만 유지해줘`
2) 구현 또는 분석
   - 기능/버그/성능: `@delivery-engineer ...`
   - 로컬 에러: `@local-error-fixer ...`
   - 성장/SNS: `@growth-operator ...`
3) 안전 검증
   - `@safety-verifier 방금 변경 파일 기준으로 lint/typecheck/test와 정책 체크를 실행해줘`
4) 종료 기록
   - `@plan-coordinator 실행 결과/검증/블로커를 PROGRESS에 기록하고 PLAN 상태를 갱신해줘`

## 3.1 금지 규칙
- `@delivery-engineer`가 검증 없이 완료 선언하는 것 금지
- `@safety-verifier`가 수정까지 하는 것 금지(read-only 유지)
- `@growth-operator`가 근거 없는 채널 주장만 하는 것 금지(출처/신뢰도 필수)
- `@local-error-fixer`가 에러와 무관한 리팩터를 확장하는 것 금지

## 4) 에이전트별 정확한 사용법

### 4.1 `@delivery-engineer`
언제 쓰나:
- 기능 추가, API/DB 변경, 버그 수정, 성능 개선

작업 모드(반드시 1개 선택):
- Feature: Prisma -> Zod -> Service -> Action/Route -> UI -> Tests
- Bugfix: repro -> root cause -> minimal patch -> regression test
- Performance: baseline(p50/p95) -> bottleneck 1개 -> patch -> remeasure
- Maintenance: 계약 변화 없는 내부 정리

입력 템플릿:
```text
@delivery-engineer
목표: [무엇을 바꿀지]
범위: [파일/레이어]
완료기준: [테스트/동작/지표]
제약: [건드리면 안 되는 것]
모드: [Feature|Bugfix|Performance|Maintenance]
```

반드시 기대할 출력:
- 적용한 모드(Feature/Bugfix/Performance/Maintenance)
- root cause 또는 설계 근거(2~4줄)
- 변경 파일 목록
- 실행한 검증 명령 + 결과
- 남은 리스크 + 다음 1개 명령

### 4.2 `@safety-verifier`
언제 쓰나:
- 코드/정책/마이그레이션 변경 후 머지 전 검증

입력 템플릿:
```text
@safety-verifier
검증대상: [변경 파일 또는 기능]
필수게이트: lint, typecheck, targeted test
정책체크: rate-limit/new-user/report-auto-hide/link-contact/local-vs-global
릴리스체크: migration rollback note/health/log/monitoring 영향
```

반드시 기대할 출력:
- `PASS | PASS_WITH_WARNINGS | FAIL`
- Blocker/Warning 분리
- 실패 시 첫 실패 명령 + 최소 재현
- Go/No-Go 권고

### 4.3 `@local-error-fixer`
언제 쓰나:
- Next.js/TypeScript/build/runtime 에러 로그가 있을 때

입력 템플릿:
```text
@local-error-fixer
에러로그: [원문 붙여넣기]
재현명령(선택): [예: pnpm typecheck]
범위제한: [수정 가능한 파일/영역]
```

반드시 기대할 출력:
- 파싱된 에러 요약
- 루트 원인
- 최소 수정 파일
- 재검증 결과
- 잔여 리스크(있으면 1~2개)

### 4.4 `@growth-operator`
언제 쓰나:
- 초기 유저 유입, SNS 실행, 채널 실험 설계

기본 채널:
- Naver Blog/Search
- Kakao Open Chat/Community
- Instagram Reels/Feed

커뮤니티 유즈케이스 기본 세트:
- 동네 질문/후기 참여 유도(읽기 -> 댓글 -> 재방문)
- 실종/목격 공용보드 신뢰 유입(긴급성 기반 유입)
- 품종 라운지 반복 방문(관심사 구독형)
- 병원/거래 공용보드 안전 메시징(정책 준수형 카피)

입력 템플릿:
```text
@growth-operator
목표: [예: 14일 내 가입 전환율 개선]
제약: [1인 운영/무광고/익명성 우선]
산출물: 7/14/30 실행안 + 채널별 카피 + KPI go/stop
```

반드시 기대할 출력:
- 근거표(출처/신뢰도 포함)
- 7/14/30 실행 보드
- 채널별 카피 템플릿
- KPI 컷오프와 keep/fix/kill
- 법적/정책 리스크 제어 문구

### 4.5 `@orchestrator`
언제 쓰나:
- 요청이 혼합되어 어느 에이전트로 시작할지 불명확할 때

입력 템플릿:
```text
@orchestrator
요청: [혼합 요청 원문]
요구: lane 분류 + 실행 순서 + 담당 에이전트 지정
```

반드시 기대할 출력:
- primary lane
- 실행 순서
- 각 단계 owner + next command + completion condition
- 혼합 요청인 경우 분할 기준(왜 나눴는지)

### 4.6 `@plan-coordinator`
언제 쓰나:
- 세션 시작/종료, PLAN/PROGRESS 반영, 핸드오프 기록

입력 템플릿:
```text
@plan-coordinator
작업: [시작 기록|종료 기록|상태 갱신]
증빙: [실행한 명령/결과]
반영범위: [PLAN, PROGRESS]
```

## 5) 실전 예시

### 기능 작업
```text
@delivery-engineer
목표: 공용보드 목록 API에 IP rate-limit 적용
범위: boards/communities route + contract test
완료기준: targeted test/lint/typecheck 통과
```

### 검증 작업
```text
@safety-verifier
검증대상: 공용보드 목록 API 변경분
필수게이트: targeted lint/typecheck/test
정책체크: rate-limit 회귀 여부
```

### 로컬 에러 수정
```text
@local-error-fixer
에러로그: Next.js build failed: Cannot find module '@/lib/env' ...
```

### 성장 작업
```text
@growth-operator
목표: 온동네 초기 30일 유입 운영안
제약: 1인 운영, 무광고, 익명성 우선
산출물: 채널별 실행 캘린더 + KPI go/stop
```

### 혼합 요청 분해
```text
@orchestrator
요청: 공용보드 정책 수정과 인스타 유입 문구를 같이 정리
요구: product-delivery / growth-sns-community로 분리하고 실행 순서 제시
```

## 6) 검증 체크리스트
- `opencode agent list`에서 6개 커스텀 에이전트가 보이는지 확인
- `@delivery-engineer` 직접 호출이 정상 동작하는지 확인
- `@safety-verifier`가 read-only로 보고만 하는지 확인
- `@local-error-fixer`가 에러 로그 기반으로 재현->수정->검증을 수행하는지 확인
- `.opencode/commands`가 비어 있는지 확인(커맨드 비의존 상태)

## 6.1 최소 운영 루틴 (하루 1사이클)
1) `@plan-coordinator`로 오늘 1개 목표를 `in_progress`로 설정
2) `@delivery-engineer` 또는 `@growth-operator`로 실행
3) 코드 변경이면 `@safety-verifier` 통과
4) `@plan-coordinator`로 `PROGRESS.md` 기록 + `PLAN.md` 상태 갱신
5) 사이클 단위로 `git commit` -> `git push`
6) GitHub Actions `quality-gate` 성공 확인
7) `OPS_BASE_URL=https://<운영URL> pnpm -C app ops:check:health`로 배포 health 확인
8) `git status --short`가 비어 있고 언트랙 파일이 없는지 확인한 뒤 다음 사이클 시작

종료 기준:
- `commit -> push -> quality-gate success -> ops:check:health -> worktree clean`
- 다음 사이클은 반드시 clean worktree에서 시작

## 6.2 프롬프트 자동 생성 루틴 (plan-coordinator 연계)
1) 프롬프트 초안 생성
   - `pnpm -C app agent:prompt --mode policy --goal "..." --scope "prisma,zod,service,api,tests" --policy-impact "..." --out /tmp/agent-task.txt`
2) 계획 반영
   - `@plan-coordinator`에 목표/완료기준을 먼저 등록하고 대상 cycle을 `in_progress`로 변경
3) 실행
   - 생성된 프롬프트 블록을 `@delivery-engineer`(또는 `@growth-operator`) 입력 템플릿으로 사용
4) 검증/종료
   - `@safety-verifier` 결과를 바탕으로 `@plan-coordinator`가 `PLAN.md`/`PROGRESS.md`를 최종 동기화
