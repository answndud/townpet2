# Deployment Security Checklist

기준일: 2026-03-06  
대상: TownPet production 배포 전/후 보안 점검

이 문서는 아래 두 정보를 합쳐 정리합니다.

- 현재 코드/워크플로우 기준으로 실제 필요한 secret
- 운영자가 Vercel / GitHub Actions 화면에서 직접 확인했다고 공유한 현재 설정값

주의:
- 아래 `현재 설정`은 사용자가 공유한 콘솔 목록 기준입니다.
- 내가 Vercel / GitHub UI를 직접 조회해 검증한 값은 아닙니다.

## 1. 한눈에 보기

현재 strict preflight 기준 핵심 결론:

- GitHub Actions secrets는 현재 워크플로우 기준으로 충족 상태
- Vercel production은 공유된 목록 기준으로 `UPSTASH_REDIS_REST_URL`만 확인이 더 필요함
- 나머지 fail 4건은 "키 자체 없음"보다 "로컬 env 기준 fail", "Production 슬롯 값/강도 문제" 가능성을 먼저 봐야 함
- production Vercel 배포는 `build:vercel` 초반에 동일 strict preflight를 자동 실행하므로, 필수 env 누락은 실제 배포도 fail-fast 됩니다.
- production Vercel 배포는 같은 시점에 `prisma generate`와 auth email readiness preflight도 자동 실행하므로, case-insensitive email migration readiness 문제가 있으면 `prisma migrate deploy` 전에 중단됩니다.

strict preflight 명령:

```bash
cd /Users/alex/project/townpet/app && pnpm ops:check:security-env:strict
```

auth email readiness preflight 명령:

```bash
cd /Users/alex/project/townpet/app && pnpm ops:check:auth-email-readiness
```

정상 기준:

- `summary: pass=5, warn=0, fail=0`

## 2. Vercel Production 최종 필수 목록

아래는 현재 코드 기준으로 production에 있어야 하는 최종 목록입니다.

| Key | 필수 여부 | 이유 |
|---|---|---|
| `AUTH_SECRET` 또는 `NEXTAUTH_SECRET` | 필수 | 인증 세션/JWT 시크릿 |
| `DATABASE_URL` | 필수 | 앱 runtime DB 연결 |
| `APP_BASE_URL` | 필수 | 사이트 절대 URL/메일 링크/health 기준 URL |
| `CSP_ENFORCE_STRICT=1` | 필수 | strict CSP 강제 |
| `GUEST_HASH_PEPPER` | 필수 | 비회원 식별 해시 보호 |
| `HEALTH_INTERNAL_TOKEN` | 필수 | 내부 health diagnostics 보호 |
| `UPSTASH_REDIS_REST_URL` | 필수 | rate-limit/query-cache backend |
| `UPSTASH_REDIS_REST_TOKEN` | 필수 | rate-limit/query-cache backend |
| `BLOB_READ_WRITE_TOKEN` | 운영상 사실상 필수 | 프로덕션 이미지 업로드 |
| `AUTH_SECRET` 강도 | 필수 조건 | 32자 이상 랜덤값 |
| `ENABLE_SOCIAL_DEV_LOGIN=0` 또는 미설정 | 필수 정책 | dev social login 우회 차단 |

운영 기능에 따라 필요한 값:

| Key | 필요 조건 | 이유 |
|---|---|---|
| `KAKAO_CLIENT_ID` | 카카오 로그인 사용 시 | OAuth |
| `KAKAO_CLIENT_SECRET` | 카카오 로그인 사용 시 | OAuth |
| `NAVER_CLIENT_ID` | 네이버 로그인 사용 시 | OAuth |
| `NAVER_CLIENT_SECRET` | 네이버 로그인 사용 시 | OAuth |
| `RESEND_API_KEY` | 메일 인증/재설정 운영 시 | 이메일 전송 |
| `SENTRY_DSN` | 에러 모니터링 운영 시 | Sentry 전송 |
| `CORS_ORIGIN` | cross-origin API 허용 필요 시 | CORS |

있어도 되지만 현재 필수는 아닌 값:

| Key | 현재 판단 |
|---|---|
| `NEXTAUTH_URL` | 호환용/중복 허용 |
| `DIRECT_URL` | 현재 앱 runtime에서는 미사용 |

- [app/.env.production.example](/Users/alex/project/townpet/app/.env.production.example)

## 5. strict preflight fail=5 기준 슬롯 확인 순서

현재 strict preflight 결과:

- `AUTH_SECRET_OR_NEXTAUTH_SECRET`
- `CSP_ENFORCE_STRICT`
- `GUEST_HASH_PEPPER`
- `HEALTH_INTERNAL_TOKEN`
- `UPSTASH_REDIS_REST_URL_AND_TOKEN_PAIR`

중요:
- 이 결과는 로컬에서 production 규칙으로 실행한 결과입니다.
- Vercel Production에 값이 있어도 로컬에는 없으면 fail이 날 수 있습니다.

가장 효율적인 확인 순서는 아래입니다.

### Step 1. Vercel `Production` 슬롯부터 확인

가장 먼저 확인할 슬롯:

- `Production`

확인 이유:

- strict preflight는 production 기준으로 검사함
- Preview / Development에 값이 있어도 Production에 없으면 실패

먼저 봐야 할 키 순서:

1. `UPSTASH_REDIS_REST_URL`
2. `AUTH_SECRET`
3. `CSP_ENFORCE_STRICT`
4. `GUEST_HASH_PEPPER`
5. `HEALTH_INTERNAL_TOKEN`

### Step 2. `AUTH_SECRET`는 "존재"보다 "강도" 확인

확인 포인트:

- 32자 이상 랜덤 문자열인지
- `test`, `example`, `changeme`, `dev-local-nextauth-secret` 같은 패턴이 없는지

즉:

- 키가 보여도 값이 약하면 strict preflight는 실패할 수 있음

### Step 3. `CSP_ENFORCE_STRICT` 값 확인

확인 포인트:

- Production 슬롯에서 값이 정확히 `1` 또는 `true`인지
- Preview에만 있고 Production에 없는 상태가 아닌지

### Step 4. `GUEST_HASH_PEPPER`, `HEALTH_INTERNAL_TOKEN` 값 확인

확인 포인트:

- 키가 Production 슬롯에 실제 존재하는지
- 빈 문자열이 아닌지
- Preview / Development에만 들어간 상태가 아닌지

### Step 5. `UPSTASH_REDIS_REST_URL` / `TOKEN` 쌍 확인

확인 포인트:

- `TOKEN`만 있고 `URL`이 없는지
- 둘 다 Production 슬롯에 있는지
- 프로젝트가 다른 Upstash 인스턴스를 가리키고 있지 않은지

현재 공유 목록상 가장 직접 재확인할 항목:

- `UPSTASH_REDIS_REST_URL`

### Step 6. 그 다음 `Preview` 슬롯 확인

확인 이유:

- production은 통과해도 preview에서 dev login, CSP, health token이 다르게 동작할 수 있음
- staging/preview에 실데이터를 붙여 쓰면 보안상 사고가 날 수 있음

Preview에서 권장 확인 항목:

- `ENABLE_SOCIAL_DEV_LOGIN`이 비어 있거나 `0`인지
- `AUTH_SECRET`가 production과 동등한 강도인지
- `HEALTH_INTERNAL_TOKEN`이 필요한 preview라면 존재하는지

### Step 7. 마지막으로 `Development` 슬롯 확인

확인 이유:

- 로컬/개발 편의값이 production에 잘못 복사되는 사고를 막기 위해

Development에서 확인할 것:

- dev-only 값이 Production에 섞여 올라가지 않았는지
- `ENABLE_SOCIAL_DEV_LOGIN=1`이 필요하다면 Development/CI에서만 쓰는지

## 6. 바로 실행할 체크 절차

1. Vercel Project Settings -> Environment Variables -> `Production` 필터 선택
2. 아래 5개 키 존재/값 확인
   - `AUTH_SECRET`
   - `CSP_ENFORCE_STRICT`
   - `GUEST_HASH_PEPPER`
   - `HEALTH_INTERNAL_TOKEN`
   - `UPSTASH_REDIS_REST_URL`
   - `UPSTASH_REDIS_REST_TOKEN`
3. 필요 시 `RESEND_API_KEY`도 확인
4. 값 수정 후 재배포
5. strict preflight 실행

```bash
cd /Users/alex/project/townpet/app && pnpm ops:check:security-env:strict
```

6. 배포 후 public/internal health smoke 확인

```bash
curl -sS -i https://<your-domain>/api/health
```

```bash
cd /Users/alex/project/townpet/app && OPS_BASE_URL=https://<your-domain> OPS_HEALTH_INTERNAL_TOKEN=<HEALTH_INTERNAL_TOKEN> pnpm ops:check:health
```
